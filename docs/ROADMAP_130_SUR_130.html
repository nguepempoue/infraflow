Principes de réussite

  • Toujours itérer avec des preuves : chaque étape se termine par une commande de vérification (curl / kubectl get / job CI vert).
  • Zéro secrets en dur : ni dans Dockerfile, ni YAML, ni scripts, ni workflow. Uniquement GitHub Secrets et/ou variables.
  • Pas de :latest en Kubernetes : images taguées avec SHA (ou version explicite).
  • Non-root partout : conteneurs non root + Kubernetes securityContext.runAsNonRoot: true.
  • Préparer la démo : un “happy path” clair (Docker Compose) + un “happy path” clair (Kubernetes via script).

Outils locaux

2) Dockerfiles + docker-compose

2.1 Choix des services

  • api-service : service HTTP de test type httpbin (répond à /get).
  • web-service : Nginx en reverse proxy vers api-service + endpoint “status” (ex : /status).

2.2 Dockerfiles (qualité + sécurité)

  • Base officielle et légère (alpine recommandé).
  • Non-root : utilisateur dédié + port non-privilégié si nécessaire.
  • .dockerignore : exclure .git, artefacts, node_modules, caches, etc.

2.3 docker-compose.yml (démarrage en 1 commande)

  • Réseau personnalisé (ex : infraflow-net).
  • depends_on avec condition “service_healthy”.
  • Healthcheck sur api-service (HTTP GET /get) + sur web-service (HTTP GET /get via proxy).
  • Exposer web-service sur localhost:8080.
  • Variables dans .env (fournir .env.example).

2.4 Vérifications (recette Docker)

docker compose up -d
docker compose ps
curl -i http://localhost:8080/get

Attendu : réponse HTTP 200 + JSON de httpbin (ou équivalent documenté).

3) CI/CD GitHub Actions (lint, build, scan, push)

Workflow : .github/workflows/ci-cd.yml. Triggers : push main + pull_request.

3.1 Job lint-and-test

  • hadolint sur Dockerfiles
  • lint YAML (workflow + k8s + compose)
  • shellcheck sur scripts/deploy.sh

3.2 Job build

  • Build des 2 images.
  • Tag basé sur le SHA : exemple sha-${{ github.sha }}.
  • Activer le cache docker si possible.

3.3 Job security-scan

  • Scan Trivy sur les images.
  • Fail si vulnérabilités CRITICAL (exit-code = 1).
  • Publier un rapport (artefact du job).

3.4 Job push (uniquement sur main)

  • Conditionner le job (ou le step) sur refs/heads/main.
  • Authentification au registry via secrets GitHub.

3.5 Vérifications

  • Sur une PR : lint + build + scan doivent tourner.
  • Sur main : push doit publier les images taguées SHA.

4) Kubernetes (manifestes et bonnes pratiques)

4.1 Cluster local

  • Démarrer minikube ou kind.
  • S’assurer que le cluster est accessible via kubectl.
kubectl cluster-info
kubectl get nodes

4.2 Namespace

  • Créer infraflow (manifest namespace.yml).

4.3 Deployments (api-service + web-service)

  • replicas: 2 minimum.
  • resources : requests + limits CPU/mémoire.
  • probes : readiness + liveness (HTTP).
  • securityContext : runAsNonRoot: true (et paramètres associés si besoin).
  • image tag : sha du commit (pas de latest).

4.4 Services

  • api-service : ClusterIP.
  • web-service : NodePort ou LoadBalancer (minikube peut exposer via minikube service).

4.5 Nginx ConfigMap

  • Mettre nginx.conf dans un ConfigMap.
  • Monter le ConfigMap dans le pod web-service (volume + volumeMount).

4.6 HPA sur api-service

  • CPU threshold : 70%.
  • S’assurer que Metrics Server est présent (sinon HPA restera inconnu).
kubectl get hpa -n infraflow

4.7 Déploiement + vérifications

kubectl apply -f k8s/
kubectl get pods -n infraflow
kubectl get svc -n infraflow

Attendu : pods Running, 2 replicas minimum, services présents, pas de CrashLoop.

5) Monitoring (Helm, Prometheus, Grafana, alerting)

5.1 Installer kube-prometheus-stack via Helm

helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update
helm install monitoring prometheus-community/kube-prometheus-stack --namespace monitoring --create-namespace

5.2 Vérifier l’état

kubectl get pods -n monitoring

Attendu : Grafana/Prometheus/Alertmanager en Running.

5.3 Accès local à Grafana

kubectl port-forward -n monitoring svc/monitoring-grafana 3000:80
  • Ouvrir http://localhost:3000.
  • Vérifier qu’un dashboard affiche CPU/mémoire des pods du namespace infraflow (dashboards “Kubernetes / Compute Resources / Namespace” ou équivalent).

5.4 Alerting (PrometheusRule)

  • Ajouter au moins 1 règle d’alerte réaliste (ex : pod down > 1 min) dans le cluster.
  • Vérifier qu’elle apparaît dans Prometheus / Alerts.

6) Script Bash scripts/deploy.sh

But : une commande unique qui prépare et déploie l’infra Kubernetes + affiche comment accéder au web-service et à Grafana.

6.1 Exigences minimales

  • Shebang : #!/usr/bin/env bash
  • set -euo pipefail
  • Vérification des prérequis : kubectl / helm / docker
  • Création namespace si absent
  • Apply manifestes
  • Attente rollouts : kubectl rollout status
  • Affichage URLs/commandes d’accès (port-forward / minikube service)

6.2 Vérification
